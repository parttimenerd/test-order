# test-order Dashboard — Concept & Design

## Concept

test-order is a Maven/Gradle plugin that learns which tests cover which source
classes (via bytecode instrumentation), then reorders test execution to run the
most relevant tests first when code changes. The plugin accumulates test history
(durations, failure scores, run-by-run outcomes) in a local TOML state file.

The **dashboard** is a zero-dependency single HTML file (`target/test-order-dashboard/index.html`)
generated by a new Maven goal: `mvn test-order:dashboard`. Opening it in any browser
gives a complete visual picture of the test suite's current prioritization state.

---

## Design Principles

1. **Answer "why is this test first?" in one click.** The primary user need is understanding
   the ordering. Score breakdown must be front and centre — not buried behind tabs.
2. **Read-only.** The dashboard never modifies state files, weights, or project config.
   All what-if features are purely client-side.
3. **Scannable first, deep-dive second.** The landing view should surface the most
   important signals (top-ranked tests, APFD, failure count) without requiring any clicks.
4. **No dead pixels.** If data isn't available (no coverage, no runs, no changes), hide
   the corresponding section entirely rather than showing an empty chart.
5. **Works on file://.** No server needed. No CORS issues. CDN scripts are optional
   enhancement — the dashboard must degrade to inlined libraries gracefully.
6. **Small test suites are the common case.** Optimise UX for 10–200 test classes.
   Handle 1 000+ gracefully but don't design the primary layout for that scale.
7. **Minimal chart count.** Prefer a single well-designed chart over four mediocre ones.
   Every chart must answer a specific question.

## Glossary

| Term | Definition |
|------|-----------|
| **APFD** | Average Percentage Faults Detected. Range 0–1. Higher means failures were found earlier in the test run. 1.0 = all failures ran first. |
| **EMA** | Exponential Moving Average. Smooths duration measurements; recent runs weighted more heavily via `durationAlpha`. |
| **Failure score** | A per-test-class number (≥ 0) representing recent failure frequency. Decayed by `failureDecay` each run; higher = failed more recently/often. Pruned below `failurePruneThreshold`. |
| **Speed ratio** | A per-test value in [−1, 1]. Negative = faster than median, positive = slower. Computed as `log₂(duration / median)` clamped to [−3, 3] then divided by 3. |
| **Dep overlap** | Count of source classes (or members) that a test exercises AND that have changed since last learn run. |
| **Set-cover bonus** | When `coverageBonus > 0`, replaces per-test overlap scoring with a greedy algorithm that awards declining bonuses (× 0.8) to tests greedily covering all changed classes. |
| **Changed class** | A source class whose bytecode hash differs from the learned snapshot. |
| **Dependency index** | Binary file (`.test-order/test-dependencies.lz4`) mapping each test class to the source classes it exercises. V4 includes member-level deps. |
| **State file** | `.test-order/state.lz4` (LZ4-compressed TOML). Stores durations, failure scores, run history, and weights. |
| **Learn mode** | Instruments tests to build the dependency index. |
| **Order mode** | Uses the index + change detection to reorder tests by score. |

## User Stories

| # | As a… | I want to… | So that… |
|---|-------|-----------|---------|
| U1 | Developer | Open the dashboard and immediately see why my top test is ranked #1 | I can trust (or debug) the ordering |
| U2 | Developer | See which of my changed files caused which tests to be prioritised | I understand the dependency graph |
| U3 | Developer | Spot flaky tests (fail → pass → fail pattern) | I can fix or quarantine them |
| U4 | Developer | See if the ordering is actually improving over time (APFD going up) | I know the plugin is working |
| U5 | CI engineer | Find the slowest tests in the suite | I can split them into a separate job or optimise them |
| U6 | CI engineer | Check whether failure-recency scoring is catching regressions quickly | I can tune decay/maxFailure weights |
| U7 | Team lead | See a project-level summary (test count, failure rate, APFD) | I can assess test suite health at a glance |
| U8 | Developer | Check what-if when changing weights | I can tune scoring without re-running tests |

## Reference: `show-order` Terminal Output

The existing `mvn test-order:show-order` output gives a baseline for what data
the dashboard must surface. The dashboard is the visual equivalent of this:

```
Changed classes: [com.example.PaymentService, com.example.CurrencyConverter]
Changed test classes: [com.example.PaymentServiceTest]

  #    Test Class                Score  Deps   Fail  Changed  Duration
  —    ──────────                ─────  ────   ────  ───────  ────────
  1.   PaymentServiceTest          22     3           yes       134ms
  2.   OrderFlowTest               11     1                     312ms
  3.   CurrencyConverterTest        9     2                      45ms
  4.   AuditLogTest                 5     1                      89ms
  5.   InventoryServiceTest         1     0                     201ms
  6.   UserRegistrationTest         0     0    2.5              1340ms
  7.   ReportGeneratorTest          0     0                     4200ms
```

Key observations for dashboard UX:
- **Rank** is the execution order position (1 = runs first).
- **Score** is the total from all components combined — the dashboard must break it apart.
- **Deps** column is the dependency overlap count (how many changed classes the test touches).
- Tests with **score 0** still run — they just run last. The dashboard should visually
  distinguish "prioritised" (score > 0) from "unprioritised" (score = 0).
- **Fail score** is only non-zero for recently-failing tests. Most tests show nothing here.

## How test-order Works (Plugin Context)

```
mvn test -Dtest-order.mode=learn     # instruments tests, records class coverage
                                     # writes: .test-order/test-dependencies.lz4
                                     #         .test-order/state.lz4 (durations + failures)

mvn test -Dtest-order.mode=order     # loads index, detects changes, reorders tests
                                     # changed classes → pick affected tests → run first
                                     # appends RunRecord to .test-order/state.lz4

mvn test-order:show-order            # dry-run: prints scored test order in terminal
mvn test-order:dashboard             # generates HTML dashboard ← NEW
```

### Key Data Produced

| File | Contents | Format | Typical Size |
|------|----------|--------|-------------|
| `.test-order/test-dependencies.lz4` | Binary index: test class → source class deps; V4 includes member-level deps | LZ4-compressed radix trie + RoaringBitmaps | 100 KB – 5 MB |
| `.test-order/state.lz4` | Durations (adaptive EMA), failure history (decay-weighted), run history (≤ configurable max), scoring weights | LZ4-compressed TOML | 50 KB – 2 MB |
| `target/test-order-coverage/` | JaCoCo + Surefire reports (optional, coverage mojo) | XML/HTML | varies |

## Data Available for the Dashboard

### Dependency Index (`.test-order/test-dependencies.lz4`)

For each **test class**, a set of **source class FQCNs** it exercised during the learn run.
The V4 format also stores member-level dependencies (`depClass#methodName`), enabling
precise overlap scoring (a class only counts as overlapping if the test uses a member
that actually changed, or a static initialiser changed).

```
com.example.PaymentServiceTest  →  { com.example.PaymentService,
                                     com.example.CurrencyConverter,
                                     com.example.AuditLog, ... }
  (member-level)                →  { PaymentService#charge, CurrencyConverter#convert, ... }
com.example.OrderFlowTest       →  { com.example.OrderService,
                                     com.example.PaymentService, ... }
```

### State File (`.test-order/state.lz4`)  — LZ4-compressed TOML

**Per test class (class-level):**
- `duration` — adaptive EMA-smoothed execution time (ms); alpha adjusts based on variance
- `durationVariance` — EMA variance of durations (high = flaky timing)
- `failureScore` — decay-weighted failure history (per-run decay factor, configurable; pruned below `failurePruneThreshold`)

**Per test method (method-level, when available):**
- `duration` — method-level adaptive EMA duration
- `failureScore` — method-level decay-weighted failure score (separate decay factor)

**Per run (up to `historyMaxRuns`, default 50):**
- `timestamp` — epoch millis when this run happened
- `totalTests` — how many test classes ran
- `totalFailures` — how many test classes failed
- `firstFailurePosition` — rank position of the first failure (1 = failed first; −1 = no failures)
- `apfd` — Average Percentage Faults Detected (0–1; higher = failures found earlier = better ordering)
- Per-test outcome (`TestOutcome`):
  - `testClass`, `totalScore`, `depOverlap`, `depTotal`, `failScore`
  - `isChanged`, `isNew`, `isFast`, `isSlow`, `failed`
  - `complexityOverlap`, `speedRatio` (−1 to 1), `hasStaticFieldOverlap`

**Configuration section:**
- `failureDecay`, `methodFailureDecay` — per-run decay multipliers
- `durationAlpha`, `methodDurationAlpha` — EMA smoothing factors
- `failurePruneThreshold` — score below which failure entries are pruned
- `emaVarianceThreshold` — threshold for adaptive alpha adjustment
- `historyMaxRuns` — max run records retained

### Scoring Weights

Nine configurable integer weights controlling the scoring formula, loaded from
`default-scoring-weights.toml` and overridable via state file, user TOML file, or
system properties (`-Dtestorder.score.*`):

| Weight | Default | Range | Meaning |
|--------|---------|-------|---------|
| `newTest` | 15 | 0–50 | Flat bonus for test classes not yet in the dependency index |
| `changedTest` | 9 | 0–30 | Bonus when the test's own source code changed |
| `maxFailure` | 5 | 0–20 | Cap on the failure-recency contribution |
| `speed` | 1 | 0–10 | Max bonus for tests faster than the median (log₂ scale) |
| `speedPenalty` | 1 | 0–10 | Max penalty for tests slower than the median (log₂ scale) |
| `depOverlap` | 5 | 0–30 | Weight for `overlap / √totalDeps` dependency overlap score |
| `changeComplexity` | 3 | 0–20 | Weight for complexity-weighted overlap (same normalisation) |
| `staticFieldBonus` | 2 | 0–10 | Bonus when a test exercises a changed static field (member-level only) |
| `coverageBonus` | 0 | 0–20 | Greedy set-cover bonus; 0 = off; when on, replaces depOverlap |

### Score Formula (per test class)

```
totalScore = depOverlapComponent          # min(ceil(overlap / √totalDeps × weight), weight)
           + changeComplexityComponent    # same formula with complexity-weighted overlap
           + changedTestBonus             # weights.changedTest if isChanged
           + newTestBonus                 # weights.newTest if not in depMap
           + speedBonus / speedPenalty    # log₂-scaled, clamped to [−3, 3] then normalised
           + staticFieldBonus             # weights.staticFieldBonus if member-level overlap
           + failureRecencyComponent      # min(ceil(failureScore), weights.maxFailure)
           + setCoverBonus                # greedy declining bonus (0.8× per pick) when coverageBonus > 0
```

Speed ratio detail: `log₂(duration / median)` clamped to `[−3, 3]`, normalised to `[−1, 1]`.
Negative = faster than median. The ratio is stored in `ScoreResult.speedRatio` and `TestOutcome.speedRatio`.

### Change Detection

At `order` time the plugin detects which source classes changed since the last
learn/hash snapshot (or vs. a git branch). Change modes:
`auto | since-last-run | since-last-commit | uncommitted | explicit`.

When structural change analysis is available (`ChangedMembers`), overlap is computed
at member granularity: a class only counts if the test exercises a member that changed
(or a static initialiser changed, implicitly affecting all users).

### Coverage Data (optional)

If `mvn test-order:coverage` was run (requires JaCoCo), per-class metrics are available:
- Line / method / branch coverage percentages
- Statement counts (covered + total)
- List of test classes that cover each source class

## Mojo Design (`DashboardMojo`)

### Goal

`test-order:dashboard` — reads the dependency index, state file, and optional
coverage data, then writes a self-contained HTML file.

### Parameters

| Parameter | Default | Description |
|-----------|---------|-------------|
| `testorder.index` | `.test-order/test-dependencies.lz4` | Path to dependency index |
| `testorder.stateFile` | `.test-order/state.lz4` | Path to state file |
| `testorder.dashboard.output` | `target/test-order-dashboard/index.html` | Output HTML path |
| `testorder.dashboard.coverageDir` | `target/test-order-coverage/` | Optional coverage data directory |
| `testorder.changeMode` | `auto` | Change detection mode for scoring |
| `testorder.dashboard.open` | `false` | Auto-open in browser after generation |

### Data Flow

```
DashboardMojo.execute()
  ├─ loadState()                → TestOrderState
  ├─ loadDependencyMap()        → DependencyMap (class + member deps)
  ├─ detectChangedClasses()     → Set<String> changedClasses
  ├─ buildTestScorer()          → TestScorer (scores all tests)
  ├─ loadCoverage() [optional]  → coverage metrics map
  ├─ buildDashboardModel()      → DashboardData record (all data for JSON)
  ├─ serialize to JSON          → embedded in <script> tag
  └─ write HTML template        → single file with inlined CSS/JS + JSON blob
```

### JSON Data Model (TypeScript interfaces)

The mojo serialises this into a `<script type="application/json" id="dashboard-data">`
tag inside the HTML. The frontend parses it with `JSON.parse()`.

```typescript
/** Root object embedded in the HTML. */
interface DashboardData {
  project: ProjectInfo;
  weights: ScoringWeights;
  config: StateConfig;
  medianDuration: number;           // ms; computed over all tests with known duration
  changedClasses: string[];         // FQCNs of changed source classes
  changedTestClasses: string[];     // FQCNs of changed test classes
  tests: TestEntry[];               // all known test classes, pre-sorted by rank
  runs: RunEntry[];                 // most recent first, up to historyMaxRuns
  coverage: CoverageData | null;    // null when no JaCoCo data
}

interface ProjectInfo {
  name: string;                     // Maven artifactId
  generated: string;                // ISO-8601 timestamp (UTC)
  pluginVersion: string;            // e.g. "0.7.2"
  stateFilePath: string;            // relative path to .test-order/state.lz4
  indexFilePath: string;            // relative path to .test-order/test-dependencies.lz4
}

/** Nine scoring weights — all integers. See default-scoring-weights.toml. */
interface ScoringWeights {
  newTest: number;                  // default 15
  changedTest: number;              // default 9
  maxFailure: number;               // default 5
  speed: number;                    // default 1
  speedPenalty: number;             // default 1
  depOverlap: number;               // default 5
  changeComplexity: number;         // default 2
  staticFieldBonus: number;         // default 0
  coverageBonus: number;            // default 0
}

interface StateConfig {
  failureDecay: number;             // default 0.3, per-run multiplier
  methodFailureDecay: number;       // default 0.3
  durationAlpha: number;            // default 0.85, EMA alpha
  methodDurationAlpha: number;      // default 0.85
  failurePruneThreshold: number;    // default 0.01
  emaVarianceThreshold: number;     // default 0.35
  historyMaxRuns: number;           // default 50
}

interface TestEntry {
  name: string;                     // FQCN, e.g. "com.example.PaymentServiceTest"
  rank: number;                     // 1-based execution position
  score: number;                    // total integer score (sum of all components)

  // ── Score components (raw inputs, NOT weighted) ──────────────────
  depOverlap: number;               // count of changed classes this test touches
  depTotal: number;                 // total dependency count for this test
  failScore: number;                // decay-weighted failure score (float ≥ 0)
  speedRatio: number;               // [−1, 1]; negative = faster than median
  complexityOverlap: number;        // sum of normalised complexity values (0.0–~3.0)

  // ── Flags ────────────────────────────────────────────────────────
  isNew: boolean;                   // test class not yet in dep index
  isChanged: boolean;               // test source code changed
  isFast: boolean;                  // speed score was positive
  isSlow: boolean;                  // speed score was negative
  hasStaticFieldOverlap: boolean;   // test exercises a changed static field

  // ── Timing ───────────────────────────────────────────────────────
  duration: number;                 // EMA-smoothed duration in ms (−1 if unknown)
  durationVariance: number;         // EMA variance (−1 if unknown)

  // ── Dependencies (for graph tab) ─────────────────────────────────
  deps: string[];                   // source class FQCNs this test exercises
  memberDeps: string[] | null;      // "DepClass#member" entries; null if V1–V3 index
}

interface RunEntry {
  timestamp: number;                // epoch millis
  totalTests: number;
  totalFailures: number;
  firstFailurePosition: number;     // 0-based index; −1 if no failures
  apfd: number;                     // 0.0–1.0
  outcomes: RunOutcome[];           // one per test that ran; empty array if 0 failures
}

/** Compact per-test outcome within a run. */
interface RunOutcome {
  testClass: string;
  depOverlap: number;
  depTotal: number;
  failScore: number;
  speedRatio: number;
  complexityOverlap: number;
  isNew: boolean;
  isChanged: boolean;
  isFast: boolean;
  isSlow: boolean;
  failed: boolean;
  hasStaticFieldOverlap: boolean;
  // Note: totalScore is NOT stored in outcomes — it must be recomputed from
  // components + weights for what-if scenarios. See "Computed Fields" section.
}

/** Optional coverage data (from JaCoCo reports). */
interface CoverageData {
  classes: CoverageClass[];
}

interface CoverageClass {
  name: string;                     // FQCN
  pkg: string;                      // Java package
  lineCoverage: number;             // 0.0–1.0
  methodCoverage: number;           // 0.0–1.0
  branchCoverage: number;           // 0.0–1.0
  statementsTotal: number;
  statementsCovered: number;
  coveredByTests: string[];         // test class FQCNs
}
```

### Realistic Data Ranges

These are typical values the frontend should expect for a medium-sized project
(100–300 test classes). Use for UI layout decisions and chart axis ranges.

| Field | Typical range | Extremes | Notes |
|-------|--------------|----------|-------|
| `tests.length` | 20–300 | 1–2 000 | Most projects < 200 |
| `runs.length` | 1–50 | 0 (edge case) | Capped by `historyMaxRuns` |
| `changedClasses.length` | 0–20 | 0–500 | Often 1–5 for a single PR |
| `score` | 0–30 | 0–50 | Most tests score 0 when no code changed |
| `depOverlap` | 0–5 | 0–50 | Usually 0 for most tests |
| `depTotal` | 1–50 | 0–300 | Varies with test breadth |
| `failScore` | 0.0 | 0.0–5.0 | Vast majority are 0.0 |
| `speedRatio` | −0.5–0.5 | −1.0–1.0 | Clusters around 0 |
| `duration` | 10–500 ms | 1 ms–30 s | Log-distributed |
| `durationVariance` | 0–100 | 0–10 000 | High = flaky timing |
| `apfd` | 0.5–0.95 | 0.0–1.0 | Below 0.5 is actively bad |
| `complexityOverlap` | 0.0–1.0 | 0.0–5.0 | Usually 0 |
| `deps[].length` | 5–30 per test | 0–300 | Typical dependency set |
| `memberDeps[].length` | 10–100 per test | null or 0–2000 | null if V1–V3 index |
| `outcomes[].length` per run | 0 (no failures) | 0–2000 | Only populated if run had failures |

**Key data shape insight**: For most runs, most tests score 0 and have depOverlap 0.
The dashboard should visually emphasise the minority of tests with score > 0 (the
"hot" tests) and let the zero-score tests fade into the background.

### Computed Fields (frontend must derive)

These values are NOT in the JSON — the frontend computes them from the raw data.
This enables what-if scenarios (weight slider changes).

```typescript
/** Recompute a test's total score from raw data + weights. */
function computeScore(t: TestEntry, w: ScoringWeights, medianDuration: number): number {
  let score = 0;

  // Dep overlap: min(ceil(depOverlap / sqrt(depTotal) * w.depOverlap), w.depOverlap)
  if (t.depOverlap > 0 && t.depTotal > 0 && w.depOverlap > 0) {
    const norm = t.depOverlap / Math.sqrt(t.depTotal);
    score += Math.min(Math.ceil(norm * w.depOverlap), w.depOverlap);
  }

  // Change complexity: same formula as depOverlap but with complexityOverlap
  if (t.complexityOverlap > 0 && t.depTotal > 0 && w.changeComplexity > 0) {
    const norm = t.complexityOverlap / Math.sqrt(t.depTotal);
    score += Math.min(Math.ceil(norm * w.changeComplexity), w.changeComplexity);
  }

  // Changed test bonus
  if (t.isChanged) score += w.changedTest;

  // New test bonus
  if (t.isNew) score += w.newTest;

  // Speed: map speedRatio [-1, 1] → bonus/penalty
  // speedRatio < 0 → faster → bonus up to w.speed
  // speedRatio > 0 → slower → penalty up to w.speedPenalty
  if (t.speedRatio < 0) {
    score += Math.round(Math.abs(t.speedRatio) * w.speed);
  } else if (t.speedRatio > 0) {
    score -= Math.round(t.speedRatio * w.speedPenalty);
  }

  // Static field bonus
  if (t.hasStaticFieldOverlap && w.staticFieldBonus > 0) {
    score += w.staticFieldBonus;
  }

  // Failure recency: min(ceil(failScore), maxFailure)
  if (t.failScore > 0) {
    score += Math.min(Math.ceil(t.failScore), w.maxFailure);
  }

  // Note: set-cover bonus (coverageBonus) cannot be recomputed client-side because
  // it depends on the greedy selection order across all tests. When coverageBonus > 0,
  // the what-if slider should be disabled for depOverlap/changeComplexity/coverageBonus
  // and a note should explain "Set-cover scoring requires server-side recomputation."

  return score;
}

/** Derive per-test score component breakdown for the stacked bar chart. */
function scoreBreakdown(t: TestEntry, w: ScoringWeights): Record<string, number> {
  return {
    depOverlap: t.depOverlap > 0 && t.depTotal > 0
      ? Math.min(Math.ceil(t.depOverlap / Math.sqrt(t.depTotal) * w.depOverlap), w.depOverlap) : 0,
    changeComplexity: t.complexityOverlap > 0 && t.depTotal > 0
      ? Math.min(Math.ceil(t.complexityOverlap / Math.sqrt(t.depTotal) * w.changeComplexity), w.changeComplexity) : 0,
    changedTest: t.isChanged ? w.changedTest : 0,
    newTest: t.isNew ? w.newTest : 0,
    speed: t.speedRatio < 0 ? Math.round(Math.abs(t.speedRatio) * w.speed) : 0,
    speedPenalty: t.speedRatio > 0 ? -Math.round(t.speedRatio * w.speedPenalty) : 0,
    staticField: t.hasStaticFieldOverlap ? w.staticFieldBonus : 0,
    failureRecency: t.failScore > 0 ? Math.min(Math.ceil(t.failScore), w.maxFailure) : 0,
  };
}

/** Extract per-test history from runs array. */
function testHistory(testName: string, runs: RunEntry[]): {
  timestamps: number[];
  scores: number[];   // recomputed from outcomes + current weights
  failed: (boolean | null)[];  // null = test didn't participate in that run
  positions: (number | null)[]; // rank in that run; null if absent
} { /* iterate runs, find matching outcome */ }

/** Format FQCN for display: strip common prefix, show simple name. */
function shortName(fqcn: string): string {
  // "com.example.service.PaymentServiceTest" → "PaymentServiceTest"
  // Or with package hint: "s.PaymentServiceTest" (first letter of each package segment)
  const parts = fqcn.split('.');
  return parts[parts.length - 1];
}
```

## Framework Decision: React over Vue 3

**React** (with TanStack ecosystem) is the chosen frontend framework over Vue 3.

**Rationale**:
- Vue 3 is ~33–42 KB gzipped (core + Pinia); React + ReactDOM + Zustand is ~44–58 KB — a
  modest ~15 KB overhead that is negligible once chart/D3 libraries are factored in.
- React dominates the **headless UI ecosystem**: TanStack Table and TanStack Virtual are the
  industry standard for virtualized grids with complex sorting/filtering state.
- React's D3 integration patterns (use `useEffect` + refs, let D3 own the canvas) are
  better documented and more mature than Vue equivalents.

## Build Architecture: Vite + vite-plugin-singlefile

The dashboard compiles to **one self-contained HTML file** via a Vite build, not a CDN bundle.

### Why not CDN-only?

A CDN approach works but has weaknesses:
- External network required at open time (breaks in offline enterprise CI environments).
- No tree-shaking — all of D3 loads instead of just `d3-force` and `d3-hierarchy`.
- No TypeScript, no JSX, more fragile template code.

### Build pipeline

```
dashboard-ui/                   ← React source (not committed to main Maven module)
├── src/
│   ├── App.tsx
│   ├── components/
│   └── data/types.ts           ← TypeScript interfaces matching JSON schema
├── vite.config.ts
│   └── plugins: [react(), viteSingleFile(), tailwindcss()]
├── package.json
└── dist/
    └── index.html              ← single file (~400–600 KB, all inlined)
```

**Key Vite config**:
```typescript
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import { viteSingleFile } from 'vite-plugin-singlefile'
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  plugins: [react(), viteSingleFile(), tailwindcss()],
  build: {
    cssCodeSplit: false,
    assetsInlineLimit: 100_000_000,   // force all assets inline as data URIs
    target: 'esnext',
  },
})
```

The compiled `dist/index.html` is **checked into the Maven resource tree** at
`test-order-maven-plugin/src/main/resources/dashboard-template.html`.
The `DashboardMojo` injects the JSON data payload at a known marker and writes the
result to `target/test-order-dashboard/index.html`.

### CSP compliance mode

In secured enterprise CI environments, strict Content Security Policies block inline
scripts. When `testorder.dashboard.separateAssets=true`, the Mojo emits adjacent
`dashboard.js` and `dashboard.css` files alongside `index.html` instead of inlining
everything, matching the pattern used by Playwright's `--reporter=html`.

| **react-chartjs-2** | ~5 KB | React wrapper for Chart.js |
| **D3.js v7 (selective)** | ~30 KB (tree-shaken) | `d3-force` for dependency graph physics; `d3-hierarchy` for treemap |
| **Tailwind CSS v4** | ~5 KB (purged) | Utility-first styling via `@tailwindcss/vite` — zero CDN, statically analyzed and tree-shaken |

**Total estimated bundle**: ~350–500 KB minified (~150–200 KB gzipped).

### Virtualization: TanStack Table + TanStack Virtual

Standard `<table>` rendering breaks at ~500+ rows (50 000+ DOM nodes → layout thrashing).
TanStack Virtual renders only viewport-visible rows with `translateY` offsets, maintaining
60 FPS scroll. **Fixed row height required** — variable heights force `ResizeObserver`
overhead. Row height: `48px` (rank + name + score + badges all fit in one line).

### D3 physics constraint

Force-directed simulations are O(n log n). With `file://` blocking Web Workers, large
graphs freeze the main thread. Mitigations:
- Default to **Focus Mode** (selected test + direct deps only, ≤ 50 nodes).
- "Changed subgraph" mode: only tests touching ≥ 1 changed class (typically ≤ 100 nodes).
- Full graph auto-disabled above 300 nodes with a warning.

## Tab 1 — Test Explorer (table)

**Purpose**: Find and select tests. Understand at a glance which tests are
prioritized and why.

**Columns**: rank · name · score · dep overlap (n/total) · fail score · duration · speed ratio · badges

**Badges**: `CHANGED` (amber) · `NEW` (green) · `FAILING` (red) · `FAST` (cyan) · `SLOW` (orange) · `STATIC` (purple, has static field overlap)

**Controls**: free-text search · filter chips (badge type, score range, package) ·
sort by any column · column visibility toggle · CSV export button

**Performance**: Virtual scrolling for suites with 500+ test classes — only DOM nodes
in the viewport are rendered. The full dataset stays in memory for instant filtering.

## Tab 2 — Score Breakdown

**Purpose**: For a selected test, explain exactly why it got its score.

**Main visual**: Horizontal stacked bar chart — one segment per score component,
color-coded by type:

| Segment | Color | Source |
|---------|-------|--------|
| Dep overlap | Blue | `depOverlapScore(depOverlap, depTotal, weights.depOverlap)` |
| Change complexity | Indigo | `complexityScore(complexityOverlap, depTotal, weights.changeComplexity)` |
| Changed test | Amber | `weights.changedTest` if `isChanged` |
| New test | Green | `weights.newTest` if `isNew` |
| Speed bonus | Cyan | positive speed bucket from log₂ scale |
| Speed penalty | Orange | negative speed bucket (shown as deficit) |
| Static field | Purple | `weights.staticFieldBonus` if `hasStaticFieldOverlap` |
| Failure recency | Red | `min(ceil(failScore), weights.maxFailure)` |
| Set-cover bonus | Teal | declining greedy bonus when `coverageBonus > 0` |

**Supplemental info cards** (dynamic, only shown when the component contributes):
- "3 of 12 dependencies changed (25%) — overlap/√deps = 0.87"
- "Speed ratio: −0.3 (log₂ scale; 23% faster than median 250 ms)"
- "Failure score: 2.5 — decayed with factor 0.7 per run"
- "Complexity overlap: 0.8 (sum of normalised change sizes across 3 classes)"
- "Static field overlap: test exercises a changed static field in `PaymentService`"
- "Set-cover bonus: 5 (1st pick in greedy cover, decline factor 0.8)"

**What-if slider**: Temporarily adjust weight sliders to see how the score and
rank would change. Does not modify any files — purely client-side simulation
using the stored `speedRatio`, `depOverlap`, `depTotal`, `failScore`, and flags.

## Tab 3 — Dependency Graph

**Purpose**: Visual map of which tests cover which source classes, and which
source classes are "hot" (changed, or covered by many tests).

**Nodes**:
- Test classes → blue circles, size ∝ score
- Changed source classes → red circles
- Unchanged source classes → light gray circles, size ∝ number of tests covering them

**Edges**: test → source class it exercises (gray = unchanged dep, amber = changed dep)

**View modes** (toggle):
- **Focus** (default when test selected): selected test + its direct deps only
- **Changed subgraph**: only tests that touch ≥ 1 changed class + those changed classes
- **Full graph**: everything (auto-disabled if > 300 nodes, offers "changed" fallback)
- **Member-level** (when V4 data available): show member-level edges instead of class-level

**Interaction**: hover = tooltip (class name, dep count, score); click test node = select;
drag = pin; scroll = zoom; double-click = reset layout

**Package grouping**: Optional convex-hull overlays color-coded by package; collapses
classes within a package into a single cluster node when the graph exceeds 150 nodes.

## Tab 4 — Project Timeline

**Purpose**: See how test ordering quality and failure rates evolve over time across runs.

**4 aligned charts sharing the same time x-axis:**

1. **APFD over time** — line/area; higher = tests fail earlier in run = better ordering.
   Includes a faint band showing the expected APFD of random ordering for comparison.
2. **Failures per run** — bar chart; green = 0, red gradient for higher counts
3. **Time to first failure** — scatter; lower position = plugin found the failure sooner;
   runs with no failures shown as hollow dots at the top
4. **Test count per run** — area; sudden jumps = `learn` run added new test classes

All x-axis labels are human-readable timestamps. Hover shows a cross-hair with
values across all four aligned charts. Click a run to set it as the "selected run"
context for Tab 5.

---

## Tab 5 — Per-Test History

**Purpose**: For a selected test, show its track record across recent runs.

**4 stacked charts:**

1. **Duration trend** — line; per-run execution time; dashed horizontal = current EMA;
   shaded band = ± EMA variance
2. **Pass / Fail strip** — D3 strip chart; one colored bar per run (green = pass,
   red = fail, gray = not run); click a bar to select that run in Tab 4
3. **Score over time** — line; reflects how changed-class overlap and failure score
   varied across runs; tooltip shows component breakdown at each point
4. **Run position** — line; which rank position was this test given?
   (inverted y: 1 = top priority)

---

## Tab 6 — Distributions

**Purpose**: Understand the overall shape of the test suite — are tests mostly fast?
Mostly low-score?

**4 histograms:**

1. **Score distribution** — buckets 0–5, 5–10, …; selected test's bucket highlighted;
   annotation line showing median and mean
2. **Duration distribution** — log₂ buckets: < 10 ms, 10–100 ms, 100 ms–1 s, 1–10 s, > 10 s
3. **Dependency count** — how many source classes does each test exercise?
   Outliers (> 95th percentile) labelled
4. **Top failing tests** — horizontal sorted bar; top 20 by current failure score;
   color intensity ∝ failure score

---

## Tab 7 — Weights Explorer

**Purpose**: Understand and experiment with how scoring weights affect test ordering.

**Current weights table**: Shows the 9 weights with their current values, default
values, and optimizer ranges (min/max from `WeightDef`).

**What-if reorder**: Drag sliders for each weight; the dashboard re-scores all tests
client-side using the raw data (overlap counts, speedRatios, failScores, flags) and
shows a side-by-side comparison of old rank vs. new rank. Highlights tests that would
move significantly (Δ rank > 5). Does not write any files.

**Weight sensitivity heatmap**: For each weight, shows how much the top-10 ranking
changes when that weight varies from its min to max while others stay fixed.

---

## Tab 8 — Coverage Treemap *(optional)*

**Shown only if JaCoCo coverage data is present.**

**Purpose**: Visually identify coverage gaps — large red areas = big untested classes.

- **D3 treemap** grouped by package → class
- Rectangle **size** = `statementsTotal` (larger class = bigger area)
- Rectangle **color** = line coverage % (red = 0%, yellow = 50%, green = 100%)
- Hover tooltip: class name, line / method / branch %, test count
- Module dropdown filter for multi-module projects
- Click a class to see which tests cover it (cross-links to Test Explorer filter)

## Error & Empty States

| Condition | Behaviour |
|-----------|-----------|
| No state file (`.test-order/state.lz4` missing) | Show "No runs recorded yet" with instructions to run `mvn test-order:combined test` |
| No dependency index (`.test-order/test-dependencies.lz4` missing) | Show "No dependency data" with learn-mode instructions |
| State file but 0 run records | Show Explorer and Distributions only; Timeline/History tabs disabled with hint |
| Dependency index but 0 changed classes | All dep-overlap scores are 0; show info banner "No changes detected — scores reflect failure history and speed only" |
| Coverage data missing | Tab 8 hidden; no tab shown in nav |
| Single run only | Timeline shows single point; disable trend arrows in header KPIs |

## Performance Considerations

| Concern | Mitigation |
|---------|-----------|
| 1000+ test classes in Explorer | TanStack Virtual: fixed 48px rows, only ~20–30 DOM rows rendered; O(1) scroll position calculation |
| Large dependency graph (> 300 nodes) | Auto-switch to "changed subgraph" mode; offer explicit "load full" button with warning |
| 50 runs × 1000 tests = 50K outcome rows | Lazy-compute per-test history on selection; don't pre-build all 1000 chart datasets |
| Chart.js re-renders on tab switch | Destroy and recreate charts on tab activation; don't keep off-screen canvases |
| JSON blob > 5 MB | TOML state already LZ4-compressed; for JSON, omit member-level deps unless member-level tab is active; offer "compact" mode |
| `file://` CORS restrictions | All resources inlined or loaded from CDN with `crossorigin`; no fetch() calls to local files |

## Accessibility & Colour

- All colour-coded elements have a secondary encoding (icon, pattern, or label) so the
  dashboard is usable with colour-vision deficiency.
- Badge colours chosen from a CVD-safe palette (blue / amber / green / red that remain
  distinguishable under protanopia and deuteranopia).
- Keyboard navigation: Tab through panels and interactive elements; Enter to select a test;
  arrow keys within the Explorer table; Escape to deselect.
- ARIA labels on all chart canvases with a text summary of the data.
