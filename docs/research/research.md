# [Architectural Dossier: AI-Driven JUnit 5 Priority Order System with Dynamic Bytecode Instrumentation]{.c6 .c39} {#architectural-dossier-ai-driven-junit-5-priority-order-system-with-dynamic-bytecode-instrumentation .c28}

## [Introduction to Telemetry-Driven Test Execution Architectures]{.c16 .c6} {#introduction-to-telemetry-driven-test-execution-architectures .c28}

[In the contemporary landscape of software engineering, the exponential
growth of enterprise application repositories inevitably introduces
profound bottlenecks within continuous integration and continuous
deployment pipelines. As codebases scale into millions of lines of code,
the associated unit and integration test suites expand proportionally,
transforming test execution durations from minutes to hours. The
traditional paradigm of test execution is fundamentally agnostic to the
underlying application dependency graph. By default, testing frameworks
process test classes sequentially, alphabetically, or pseudorandomly,
completely disregarding the specific architectural vectors modified by a
given code commit.]{.c0}[1]{.c8 .c24}[ To achieve intelligent,
predictive test suite optimization---particularly through \"fail fast\"
methodologies orchestrated by artificial intelligence---a deterministic,
empirically verified mapping of test classes to their precise execution
footprints is strictly required.]{.c0}[2]{.c2}

[This technical dossier presents an exhaustive, expert-level
architectural blueprint for implementing an AI-assisted JUnit 5 Priority
Order system. The proposed architecture departs from conventional,
highly inaccurate static analysis techniques (which fail to account for
reflection, dynamic proxies, and late binding) and instead introduces a
telemetry-driven execution paradigm. This paradigm operates via a
dual-mode execution strategy, orchestrated through the programmatic
manipulation of the Maven Object Model at runtime. A custom Maven plugin
dynamically injects an instrumenting Java agent based on the
dead-code-agent framework, conducting deep bytecode manipulation to
trace active class initializations during test execution.]{.c0}[4]{.c2}

[To process and store the massive telemetry data generated by this
dynamic analysis without degrading build performance, a highly
optimized, hybrid compressed data format is engineered. This format
fuses a Radix Trie dictionary with Roaring Bitmaps, establishing an
extremely memory-efficient, serialized relational matrix between test
classes and their underlying execution dependencies.]{.c0}[7]{.c8
.c24}[ Ultimately, a custom JUnit 5 ClassOrderer ingests this binary
telemetry to dynamically reorder test classes prior to execution,
applying heuristic intersections to prioritize tests most likely to fail
based on recent code mutations.]{.c0}[1]{.c2}

## [Dynamic Maven Plugin Engineering and Runtime Orchestration]{.c6 .c16} {#dynamic-maven-plugin-engineering-and-runtime-orchestration .c28}

[The orchestration of the dependency telemetry pipeline requires deep,
non-invasive integration with the Maven build lifecycle. Relying on
developers to manually configure the maven-surefire-plugin to attach
Java agents, manage specific classpaths, and toggle fork configurations
introduces high friction and unacceptable margins for human error.
Furthermore, manual configuration is inherently static, whereas the
architectural requirement dictates the support of two distinct
operational modes: a telemetry-gathering \"Learn Mode\" and an optimized
\"Normal Mode.\" Consequently, a custom Maven plugin (MOJO) is designed
to programmatically locate and modify the Surefire configuration at
runtime, intercepting the build process before the test phase is
invoked.]{.c0}[4]{.c2}

### [Manipulating the Surefire Configuration Object Model]{.c27 .c6} {#manipulating-the-surefire-configuration-object-model .c28}

[During the Maven build execution, specifically bound to the
process-test-classes or initialize phase, the custom MOJO interrogates
the active MavenProject instance to retrieve all mapped plugins utilized
in the current build context.]{.c0}[4]{.c8 .c24}[ Upon isolating the
maven-surefire-plugin by its group and artifact identifiers
(org.apache.maven.plugins:maven-surefire-plugin), the MOJO accesses its
runtime configuration. In the Maven API, this configuration is
instantiated as an Xpp3Dom object, which provides an XML-like

### [Automated Agent Artifact Resolution]{.c27 .c6} {#automated-agent-artifact-resolution .c28}

[The automation of the agent attachment necessitates that the Java agent
artifact is physically present on the build system prior to Surefire
forking the JVM.]{.c0}[17]{.c8 .c24}[ The custom MOJO employs a
non-destructive attachment strategy via Maven\'s dependency resolution
capabilities. During the initialization phase, the MOJO utilizes the
Maven Artifact Resolver to download the telemetry-agent.jar from the
central or corporate repository. It then stages this artifact by copying
it to a fixed, absolute path within the current build directory (e.g.,
\${project.build.directory}/agents/telemetry-agent.jar).]{.c0}[17]{.c8
.c24}[ By injecting this absolute path into the dynamically constructed
\<argLine\>, the MOJO completely isolates the user from the complexities
of JVM instrumentation. The user does not need to declare the agent as a
project dependency, nor manually configure staging paths.]{.c1 .c0}

## [Operational Paradigms: State Isolation vs. Optimized Execution]{.c16 .c6} {#operational-paradigms-state-isolation-vs.-optimized-execution .c28}

[The custom MOJO serves as the central switchboard for the dual-mode
operational paradigm, toggling between telemetry acquisition and
accelerated test execution based on a user-defined system property
switch (e.g., -Dtelemetry.mode=learn). The configuration of JVM forking
is the critical mechanism differentiating these two
modes.]{.c0}[19]{.c2}

### [Learn Mode: Ensuring Pristine Static State Isolation]{.c6 .c27} {#learn-mode-ensuring-pristine-static-state-isolation .c28}

[In Learn Mode, the absolute primary objective is to map every
individual test class to the exact application classes it initializes. A
profound technical hurdle in this dynamic analysis is the mutation of
global static state within the Java Virtual Machine. If multiple test
classes execute sequentially within the same JVM instance, static
singletons, database connection pools, Spring Application Contexts, and
classloader caches initialized by the first test class will persist in
memory for the duration of the JVM\'s lifespan.]{.c0}[19]{.c2}

[According to the Java Language Specification, a class is only
initialized once per classloader. Consequently, subsequent test classes
running in the same JVM will not trigger the static initializers of
those shared dependencies, because the JVM recognizes the classes as
already loaded and initialized.]{.c0}[5]{.c8 .c24}[ This phenomenon
leads the telemetry agent to severely under-report the dependencies of
later tests, creating a corrupted, sparse mapping that would render the
AI\'s predictive ordering useless.]{.c1 .c0}

[To guarantee pristine static state isolation and ensure that every test
class triggers a full class initialization sequence for its
dependencies, the MOJO programmatically forces the Surefire
\<reuseForks\> property to false and sets \<forkCount\> to
1.]{.c0}[19]{.c8 .c24}[ Setting reuseForks=false forces Maven Surefire
to completely tear down the JVM process at the end of each test class
execution, spawning a fresh, clean JVM process with untouched state for
the next test class in the queue.]{.c0}[19]{.c8 .c24}[ This approach
entirely bypasses problems related to mutating global state and
singleton poisoning.]{.c0}[19]{.c2}

### [Normal Mode: Maximizing Throughput and Parallelism]{.c27 .c6} {#normal-mode-maximizing-throughput-and-parallelism .c28}

[The heavy performance penalty incurred by the continuous JVM teardown
and startup sequence in Learn Mode implies that this mode is
computationally expensive.]{.c0}[19]{.c8 .c24}[ It is designed to be
scheduled periodically---such as during nightly continuous integration
builds---to refresh the telemetry index, rather than on every local
developer compile cycle.]{.c1 .c0}

[In Normal Mode, telemetry acquisition is disabled. The MOJO skips the
injection of the Java agent into the argLine. Instead, its primary
function is to inject the configuration that enables the custom
AI-driven ClassOrderer. To maximize execution speed and overall
throughput, \<reuseForks\> is restored to true (or left to its default
value), allowing the JVM to be reused across numerous test
classes.]{.c0}[20]{.c8 .c24}[ Furthermore, parallel execution
configurations (such as setting \<forkCount\> to multiple CPU cores,
e.g., 2.5C) are fully supported and uninhibited, as the isolation
constraints of Learn Mode are no longer necessary.]{.c0}[20]{.c2}

## [Deep JVM Telemetry via Bytecode Instrumentation]{.c16 .c6} {#deep-jvm-telemetry-via-bytecode-instrumentation .c31}

[At the core of the dependency mapping apparatus is the custom Java
Agent, which must trace class usage with extreme precision while
maintaining a footprint that does not induce timeout failures during the
heavy I/O operations of test suites. The agent design is directly
derived from the methodology of the dead-code-agent, utilizing the
Javassist library for dynamic bytecode manipulation.]{.c0}[5]{.c2}

### [The Flaws of Standard Classloading Telemetry]{.c27 .c6} {#the-flaws-of-standard-classloading-telemetry .c28}

[Traditional classloading telemetry, such as that provided by the
standard -verbose:class JVM flag or simple ClassLoader interception, is
fundamentally flawed for the purpose of predictive dependency mapping.
The Java Virtual Machine frequently and eagerly loads classes into
memory during bytecode verification, reflection scanning, or generic
type resolution, even if those classes are never actually executed by
the program path.]{.c0}[5]{.c8 .c24}[ If the system relied merely on
class loading events, the dependency graph would be saturated with false
positives, degrading the precision of the AI\'s heuristic scoring. To
map a test to its true, actionable dependencies, the agent must record
when a class is actively used.]{.c1 .c0}

### [Instrumentation of Static Initializers]{.c27 .c6} {#instrumentation-of-static-initializers .c28}

[The telemetry agent intercepts the class loading process by
implementing a ClassFileTransformer interface, which is registered via
the Instrumentation API within the agent\'s premain method.]{.c0}[5]{.c8
.c24}[ As the JVM loads class files, the transformer is invoked,
allowing it to inspect and modify the raw bytecode before the class is
defined in memory.]{.c0}[5]{.c2}

[Utilizing the Javassist library, the transformer isolates the static
initializer block (the \<clinit\> method) of every application
class.]{.c0}[5]{.c8 .c24}[ According to the Java Language Specification
(JLS §12.4.1), a class or interface is strictly initialized only upon
its first \"active use\".]{.c0}[5]{.c8 .c24}[ Active use is formally
defined as the creation of an instance of the class, the invocation of a
static method declared by the class, or the assignment/usage of a
non-constant static field declared by the class.]{.c0}[5]{.c2}

[The Javassist transformer injects a specialized static method call
directly into the \<clinit\> block. The injected bytecode structurally
resembles the following logic:]{.c1 .c0}

## [Synchronizing JVM State with the JUnit 5 Event Model]{.c16 .c6} {#synchronizing-jvm-state-with-the-junit-5-event-model .c28}

[A profound technical challenge arises from the architectural separation
of concerns between the JVM tier and the testing framework tier. While
the Java agent possesses total awareness of ]{.c0}[which]{.c0
.c25}[ application classes are being initialized, it lacks inherent
awareness of ]{.c0}[which]{.c0 .c25}[ JUnit 5 test class is currently
triggering them. The agent operates at the absolute lowest level of the
JVM, whereas the test execution context exists entirely within the
higher-order JUnit Platform Engine.]{.c0}[27]{.c2}

[Because the Maven MOJO forces Learn Mode to operate with
reuseForks=false, every forked JVM process executes exactly one test
class before terminating.]{.c0}[19]{.c8 .c24}[ However, the agent still
requires a deterministic mechanism to extract the precise name of the
test class being executed by the current JVM in order to label the
telemetry output accurately. Parsing the command-line arguments of the
forked JVM is fragile, as Surefire frequently utilizes manifest-only
JARs or temporary boot files to bypass operating system command-line
length limitations.]{.c0}[11]{.c2}

### [Implementing the JUnit Platform TestExecutionListener]{.c27 .c6} {#implementing-the-junit-platform-testexecutionlistener .c28}

[To bridge this contextual divide, the architecture implements a custom
TestExecutionListener leveraging the JUnit Platform Launcher
API.]{.c0}[27]{.c8 .c24}[ The JUnit Platform provides a robust extension
model that allows developers to hook into the lifecycle events of the
test execution engine.]{.c0}[27]{.c2}

[The custom listener is automatically discovered and registered by the
JUnit platform via the standard Java ServiceLoader mechanism. This is
achieved by packaging a plain text file named
org.junit.platform.launcher.TestExecutionListener within the
META-INF/services/ directory of the telemetry artifact.]{.c0}[30]{.c8
.c24}[ The file contains the fully qualified class name of the custom
listener implementation, ensuring that the JUnit engine binds it during
initialization without requiring explicit annotations on the test
classes themselves.]{.c0}[30]{.c2}

[The listener implements two critical callback methods:
testPlanExecutionStarted(TestPlan) and executionFinished(TestIdentifier,
TestExecutionResult).]{.c0}[31]{.c8 .c24}[ The TestPlan object provides
a hierarchical representation of all tests discovered by the engine. As
execution commences, the listener extracts the current test class name
from the TestIdentifier object. It then utilizes a static method call or
an inter-process communication bridge to transmit this test class
identifier to the Store component initialized by the Java agent.]{.c1
.c0}

[When the test execution concludes and the JVM initiates its shutdown
sequence, a registered JVM shutdown hook triggers the serialization
routine within the agent\'s Store. The agent aggregates all application
classes recorded during the JVM\'s brief lifespan and writes out a
highly specific mapping that binds the single JUnit test class
identifier to the exhaustive set of application class dependencies. This
elegant synchronization guarantees absolute accuracy in the dependency
matrix, completely untethered from the internal complexities of the
Surefire classloader.]{.c1 .c0}

## [Hybrid Compressed Data Structure: Radix Trie and Roaring Bitmaps]{.c16 .c6} {#hybrid-compressed-data-structure-radix-trie-and-roaring-bitmaps .c28}

[As the AI telemetry system processes enterprise-scale repositories, the
generated dependency graph scales exponentially. A typical microservice
environment might contain 5,000 test classes and 15,000 application
classes. If each test class touches an average of 500 application
classes, the system generates 2.5 million string-to-string pair
mappings. Storing, parsing, and retaining this Fully Qualified Class
Name (FQCN) mapping in memory as raw JSON, XML, or plain text is
computationally prohibitive.]{.c0}[34]{.c8 .c24}[ It would inflate the
memory footprint of the build process and severely degrade the
initialization performance of the test phase before any testing even
begins.]{.c0}[34]{.c2}

[To resolve this fundamental scaling issue, a novel, hybrid compressed
data structure is engineered. This structure fundamentally separates the
storage of the string identifiers from the mathematical storage of the
relational mappings, merging the hierarchical prefix compression of a
Radix Trie with the extreme integer density of Roaring
Bitmaps.]{.c0}[8]{.c2}

### [Phase One: The Radix Trie Class Name Dictionary]{.c27 .c6} {#phase-one-the-radix-trie-class-name-dictionary .c28}

[The first phase of the compression strategy involves converting all
lengthy FQCN strings into unique 32-bit integers. Fully Qualified Class
Names in the Java ecosystem exhibit a massive degree of prefix
redundancy due to package naming conventions (e.g.,
com.enterprise.module.service.AuthService and
com.enterprise.module.service.UserService).]{.c1 .c0}

[A Radix Trie, also known as a compressed prefix tree, is utilized as
the global dictionary for this conversion.]{.c0}[8]{.c8 .c24}[ Unlike a
standard Trie where each node stores a single character (which results
in excessive pointer overhead), a Radix Trie optimizes space by merging
nodes with single children. This allows entire substrings representing
common package hierarchies (e.g., com.enterprise.module.) to be stored
in a single node.]{.c0}[8]{.c2}

[During the aggregation phase---typically executed by a Maven aggregator
plugin following the conclusion of Learn Mode---every unique application
class name encountered across all telemetry logs is inserted into the
global Radix Trie and assigned an incremental 32-bit integer ID.]{.c1
.c0}

[The Depth-First Assignment Heuristic:]{.c1 .c6}

[An advanced, third-order optimization is applied during the ID
assignment phase, representing a critical innovation in this
architecture. Rather than assigning integer IDs randomly or
chronologically as classes are encountered in the logs, the system
delays assignment until the Trie is fully constructed. It then traverses
the entire Radix Trie using a Depth-First Search (DFS) algorithm to
assign sequential integer IDs to the leaf nodes.]{.c1 .c0}

[Because the Trie naturally groups classes by their package hierarchy, a
depth-first traversal mathematically guarantees that classes residing in
the same Java package receive strictly sequential integer IDs. For
instance, all classes within the com.enterprise.core.utils package might
receive IDs from 5000 to 5049. This contiguous assignment strategy is
not merely an aesthetic grouping; it is a vital prerequisite that
unlocks extreme compression ratios in the subsequent Roaring Bitmap
phase.]{.c1 .c0}

maintaining a full Bitmap Container, it stores a single Run descriptor
pair: \[start_id, length\]. This synergy collapses the memory footprint
of massive enterprise dependency graphs by several orders of
magnitude.]{.c0}[7]{.c2}

## [Predictive Execution via the JUnit 5 ClassOrderer]{.c16 .c6} {#predictive-execution-via-the-junit-5-classorderer .c28}

[With the telemetry data securely stored and optimally compressed, the
final architectural component is the realization of the predictive test
selection and ordering logic. This is achieved through the
implementation of a custom JUnit 5 ClassOrderer.]{.c0}[1]{.c8
.c24}[ Introduced as a stable API in JUnit Jupiter 5.10, the
ClassOrderer interface dictates the global sequence in which top-level
test classes are scheduled for execution by the test
engine.]{.c0}[1]{.c2}

### [Global Registration and Property Injection]{.c27 .c6} {#global-registration-and-property-injection .c28}

[To activate the custom ClassOrderer across the entire test suite, it
must be declared via the junit-platform.properties configuration

this file in the src/test/resources/ directory leads to configuration
drift and conflicts with localized IDE testing
conventions.]{.c0}[42]{.c2}

[Instead, the custom Maven MOJO (operating in Normal Mode) automatically
synthesizes this properties file in memory and writes it to the

MOJO then ensures this directory is added to the Surefire classpath
using the \<additionalClasspathElements\> configuration.]{.c0}[29]{.c2}

### [Dynamic Order Resolution and Bitwise Intersections]{.c27 .c6} {#dynamic-order-resolution-and-bitwise-intersections .c28}

[Upon initialization by the JUnit Platform Launcher, the
PriorityClassOrderer executes a sophisticated heuristic resolution
sequence:]{.c1 .c0}

1.  [Zero-Copy Deserialization:]{.c6}[ The orderer maps the binary .idx
    file directly into memory using a MappedByteBuffer. This zero-copy
    deserialization approach entirely bypasses traditional Java I/O
    streams, instantly reconstructing the Radix Trie and the array of
    Roaring Bitmaps without allocating millions of transient
    objects.]{.c0}[40]{.c2}
2.  [AI Mutation Input Resolution:]{.c6}[ The orderer retrieves a
    predefined payload---injected by the CI/CD pipeline or an upstream
    AI analysis engine---containing a list of application FQCNs that
    have been modified in the current Git commit or pull request. This
    payload is typically supplied via an environment variable or a
    highly structured JSON artifact.]{.c0}[47]{.c2}
3.  [Dictionary Lookup:]{.c6}[ The modified FQCN strings are passed
    through the Radix Trie dictionary to rapidly retrieve their
    corresponding 32-bit integer IDs.]{.c1 .c0}
4.  [Bitwise Intersection Calculus:]{.c6}[ For every test class
    represented within the current ClassOrdererContext ]{.c0}[1]{.c8
    .c24}[, the orderer retrieves its associated Roaring Bitmap
    (representing its historical execution footprint). A lightning-fast
    bitwise AND intersection operation is performed between the test
    class\'s bitmap and the bitmap representing the modified integer
    IDs.]{.c0}[7]{.c2}
5.  [Heuristic Scoring and Sorting:]{.c6}[ A vulnerability score is
    calculated for each test class based on the cardinality (the set
    size) of the resulting intersection. The greater the overlap between
    a test\'s historical dependencies and the currently modified code,
    the higher the score. The ClassOrderer sorts the provided list of
    test classes in descending order of this heuristic score.]{.c1 .c0}

[By pushing test classes with direct dependencies on the modified code
to the absolute front of the Surefire execution queue, the system
manifests true \"fail fast\" behavior.]{.c0}[2]{.c8 .c24}[ If a recent
commit introduces a regression, the tests most likely to exercise that
specific code path are executed in the first few seconds of the build,
allowing the CI/CD pipeline to fail rapidly and provide immediate
developer feedback.]{.c1 .c0}

### [Architectural Contextualization for \@Nested Classes]{.c27 .c6} {#architectural-contextualization-for-nested-classes .c28}

[A vital architectural nuance within the JUnit 5 platform is the
hierarchical treatment of \@Nested test classes. The ClassOrderer API
explicitly defines the ordering of top-level test classes
]{.c0}[relative to each other]{.c0 .c25}[. However, inner classes
annotated with \@Nested are ordered only relative to other \@Nested
classes that share the identical enclosing parent class.]{.c0}[1]{.c2}

[Consequently, the telemetry agent and the compressed data structure
must mathematically aggregate the dependencies of all inner \@Nested
classes up to their top-level parent class representation. The Roaring
Bitmap assigned to a top-level test class represents the union of all
classes utilized by itself and its entire tree of nested descendants.
This comprehensive aggregation ensures that the parent class receives
the correct overall vulnerability score, guaranteeing that the entire
composite test structure is prioritized accurately by the
ClassOrderer.]{.c0}[1]{.c2}

## [Strategic Deployment and Cross-Module Analytics]{.c16 .c6} {#strategic-deployment-and-cross-module-analytics .c28}

[The standard Maven execution lifecycle processes multi-module reactor
builds either sequentially or in parallel.]{.c0}[20]{.c8 .c24}[ When
analyzing complex multi-module enterprise architectures, it is crucial
to recognize that the telemetry index cannot remain siloed within
individual modules; it must be aggregated globally to provide
comprehensive dependency insights.]{.c1 .c0}

[To facilitate this global intelligence, the custom Maven MOJO is
engineered to function as an aggregator plugin. While it executes
iteratively on a per-module basis during Learn Mode to attach the Java
agent and collect isolated .idx segments, a separate MOJO goal bound to
the final verify or post-integration-test phase aggregates all
individual module Radix Tries and Roaring Bitmaps into a single, master
repository-level index. This master index empowers the AI to prioritize
integration tests residing in deep downstream modules based on
foundational mutations occurring in upstream core modules.]{.c1 .c0}

[Furthermore, the integration of these telemetry paradigms introduces
significant optimizations for native image testing compilation, such as
those executed via GraalVM. Because native images cannot perform dynamic
classpath scanning or extensive reflection efficiently at runtime,
testing frameworks often struggle to discover tests. By utilizing the
UniqueIdTrackingListener paradigm ]{.c0}[32]{.c8 .c24}[, the exact
execution plan---sorted and prioritized based on the Roaring Bitmap
intersections---can be exported as a definitive text file. Passing this
explicit list of Test Unique IDs directly to the native test runner
bypasses the severe overhead of dynamic test discovery entirely, further
accelerating the verification pipeline.]{.c0}[32]{.c2}

## [Synthesis]{.c16 .c6} {#synthesis .c28}

[The architecture delineated in this dossier establishes a robust,
highly optimized, and mathematically elegant infrastructure for
AI-assisted test prioritization. By harnessing the dynamic mutability of
the Maven Object Model via Xpp3Dom, the precise, JLS-compliant bytecode
instrumentation capabilities of Javassist, the deep integration hooks of
JUnit 5 extensions, and the unparalleled compression synergies between
DFS Radix Tries and Roaring Bitmaps, organizations can deterministically
implement \"fail fast\" test execution at scale.]{.c1 .c0}

[This infrastructure seamlessly transforms historically opaque,
black-box test suites into transparent, dependency-aware pipelines. By
establishing a continuous feedback loop between static source
modifications and dynamic execution telemetry, the architecture curtails
wasted computational resources, significantly accelerates developer
feedback loops, and provides the foundational, machine-readable datasets
required for next-generation repository intelligence and predictive
artificial intelligence models.]{.c1 .c0}

## PetClinic Demo: Test Prioritization in Action

### Setup

The test-order plugin was integrated into [Spring PetClinic](https://github.com/spring-projects/spring-petclinic) (Spring Boot 4.0.3, Java 17, JUnit Jupiter 6.0.3). The project has **50 unit tests** across **11 top-level test classes**.

### Step 1: Training Run (learn mode)

```bash
./mvnw test -Dtestorder.mode=learn
```

The java agent instruments application classes at method entry and records which classes each test touches. This produced 14 `.deps` files mapping each test class to its runtime dependencies. For example, `OwnerControllerTests.deps` records dependencies on `Owner`, `OwnerController`, `OwnerRepository`, etc.

### Step 2: Baseline Order (no changes)

On the next run, the plugin auto-aggregates the `.deps` files into a `test-dependencies.idx` binary index, then activates the `PriorityClassOrderer`. With no source changes detected, all test classes score 0 and run in alphabetical order:

| # | Test Class | Score |
|---|-----------|-------|
| 1 | ValidatorTests | 0 |
| 2 | **OwnerControllerTests** | 0 |
| 3 | PetControllerTests | 0 |
| 4 | PetTypeFormatterTests | 0 |
| 5 | PetValidatorTests | 0 |
| 6 | VisitControllerTests | 0 |
| 7 | ClinicServiceTests | 0 |
| 8 | CrashControllerTests | 0 |
| 9 | I18nPropertiesSyncTest | 0 |
| 10 | VetControllerTests | 0 |
| 11 | VetTests | 0 |

### Step 3: Inject Bug in Owner.java

Modified `Owner.getAddress()` to append `" INJECTED BUG"` to the return value. This breaks `OwnerControllerTests.showOwner` and `OwnerControllerTests.initUpdateOwnerForm`, which assert `address = "110 W. Liberty St."`.

### Step 4: Order Mode with Uncommitted Changes

```bash
./mvnw test -Dtestorder.changeMode=uncommitted -Dtestorder.debug=true
```

The plugin detects `Owner.java` as modified via `git diff`, resolves it to `org.springframework.samples.petclinic.owner.Owner`, then scores each test class based on dependency overlap with the changed class. Four test classes depend on `Owner` and receive score=1 (`deps=1`). The remaining seven test classes have no dependency on `Owner` and keep score=0.

**Prioritized order:**

| # | Test Class | Score | Reason |
|---|-----------|-------|--------|
| 1 | **OwnerControllerTests** | **1** | depends on `Owner` |
| 2 | PetControllerTests | 1 | depends on `Owner` |
| 3 | VisitControllerTests | 1 | depends on `Owner` |
| 4 | ClinicServiceTests | 1 | depends on `Owner` |
| 5 | ValidatorTests | 0 | — |
| 6 | PetTypeFormatterTests | 0 | — |
| 7 | PetValidatorTests | 0 | — |
| 8 | CrashControllerTests | 0 | — |
| 9 | I18nPropertiesSyncTest | 0 | — |
| 10 | VetControllerTests | 0 | — |
| 11 | VetTests | 0 | — |

### Result

`OwnerControllerTests` moved from **position 2** (alphabetical) to **position 1** (prioritized). The 2 failing tests (`showOwner`, `initUpdateOwnerForm`) were discovered immediately as the very first tests to execute, rather than after unrelated tests.

```
[ERROR] Tests run: 13, Failures: 2, Errors: 0 -- in OwnerControllerTests
[ERROR] OwnerControllerTests.showOwner           <<< FAILURE!
[ERROR] OwnerControllerTests.initUpdateOwnerForm <<< FAILURE!
```

The "fail fast" effect is more dramatic in larger projects where the failing test class might ordinarily run as test #50 out of 500+ classes.
